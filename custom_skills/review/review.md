## 목적

작업 완료 후 사용자에게 보고하기 전에 변경된 파일을 자체 검토하고, 문제가 있으면 즉시 수정한다.
단순 질문 답변이나 파일 조회만 한 경우에는 실행하지 않는다.

## 트리거 조건

아래 중 하나라도 해당되면 보고 전에 자동 실행한다.

| 조건 | 예시 |
|------|------|
| 코드 파일 수정 또는 생성 | .py, .js, .ts, .sh, .go 등 |
| 설정 파일 수정 | .json, .yaml, .toml 등 |
| 문서 파일 수정 | .md 파일 내용 변경 |
| 스킬/에이전트/스크립트 생성 | 새 도구 파일 작성 |

## 실행하지 않는 경우

- 파일을 읽기만 한 경우
- 질문에 답변만 한 경우
- git 명령만 실행한 경우
- 이미 review 스킬이 한 번 실행된 경우 (중복 실행 방지)

## 검토 절차

### 1단계: 변경 파일 수집

현재 대화에서 Write 또는 Edit 도구로 수정한 파일 목록을 정리한다.
파일이 많으면 `git diff --name-only` 로 보완한다.

### 2단계: 항목별 검토

각 파일을 다시 읽고 파일 유형에 따라 해당하는 항목을 점검한다.

#### 공통 검토 항목

| 검토 항목 | 확인 내용 |
|-----------|-----------|
| 정확성 | 의도한 동작과 실제 코드가 일치하는가 |
| 누락 | 빠뜨린 부분이 없는가 (import, 변수 선언, 닫는 괄호 등) |
| 중복 | 기존 코드와 중복되는 로직이 있는가, 재활용 가능한 함수가 있는가 |
| 일관성 | 주변 코드의 스타일, 네이밍, 패턴과 일치하는가 |
| 오류 가능성 | null/undefined 참조, 타입 불일치, 경로 오류 등이 있는가 |
| 불필요한 코드 | 디버그 로그, 주석 처리된 코드, 사용하지 않는 import 등 |

#### 코드 파일 추가 검토 항목

| 검토 항목 | 확인 내용 |
|-----------|-----------|
| 재사용성 | 공통 유틸이나 기존 함수로 대체할 수 있는 코드가 있는가 |
| 품질 | 함수/변수 네이밍이 명확한가, 책임이 적절히 분리되어 있는가 |
| 효율성 | 불필요한 반복, 중복 연산, 비효율적인 자료구조 사용이 있는가 |

#### md 파일 추가 검토 항목

| 검토 항목 | 확인 내용 |
|-----------|-----------|
| 지시사항 준수 | 사용자가 요청한 내용이 빠짐없이 반영되었는가 |
| 완성도 | 추가로 다뤄야 할 항목이나 보완이 필요한 부분이 없는가 |
| 구조와 흐름 | 섹션 구성이 논리적이고 읽기 쉬운가, 빠진 섹션이 없는가 |
| 표현 | 모호하거나 불명확한 서술이 없는가, 예시가 필요한 곳에 예시가 있는가 |
| 형식 | 표, 목록, 코드 블록 등의 마크다운 형식이 올바른가 |

### 3단계: 즉시 수정

검토 중 발견한 문제는 사용자에게 묻지 않고 바로 수정한다.
단, 의도가 불분명한 설계 수준의 변경은 수정하지 않고 보고 시 언급한다.

### 4단계: 보고

검토 결과를 간략히 정리한다. 문제가 없었으면 별도 보고 없이 원래 작업 결과만 전달한다.

문제를 수정한 경우에만 아래 형식으로 추가 보고한다.

```
검토 후 수정 사항:
- <파일명>: <수정 내용 한 줄 요약>
```

수정하지 않았지만 사용자 판단이 필요한 항목이 있으면 별도로 언급한다.
