## code-simpler
  code-simplifier
  - 최근 수정된 코드를 자동으로 리뷰해서 가독성·일관성·유지보수성을 개선
  - 기능은 절대 바꾸지 않고, 구조와 표현만 정리
  - 주로 하는 것: 불필요한 중첩 제거, 중복 코드 정리, 변수명 개선,
  불필요한 주석 제거

  특징:
  - Opus 모델 사용 (다른 스킬보다 강력)
  - 코드 작성/수정 후 자동으로 동작하도록 설계됨
  - "적게 쓴다"보다 "명확하게 쓴다"를 우선

  현재 이 레포는 주로 md 파일이라 코드 리뷰 용도로는 잘 안 쓰일 것
  같지만, 스크립트 파일 작업할 때 유용합니다.

## claude-md-management
⏺ 두 가지 기능이 있습니다.

  1. revise-claude-md 커맨드 (/revise-claude-md)
  - 현재 세션에서 배운 것들을 CLAUDE.md 에 반영하는 용도
  - 세션 중 발견한 명령어, 패턴, 주의사항 등을 정리해서 추가 제안
  - 사용자 승인 후 적용

  2. claude-md-improver 스킬
  - 레포 전체의 CLAUDE.md 파일을 감사(audit)하는 용도
  - 각 파일을 점수(A~F)로 평가 후 품질 리포트 출력
  - 빠진 섹션, 오래된 내용, 개선 포인트 제안 → 승인 후 수정

  언제 쓰냐면:
  - 세션 끝날 때 이번에 발견한 것들을 CLAUDE.md 에 기록하고 싶을 때 →
  /revise-claude-md
  - CLAUDE.md 전반적으로 점검하고 싶을 때 → claude-md-improver

## skill-creator
### skill-creator 통해 스킬 생성
#### 실수 반복 안하게 하는 스킬 생성
❯ /skill-creator:skill-creator 클로드가 실수할때마다 특정 md 파일에 기록해두거나 기억하고  다신 같은 실수를 반복하지 않게 하고싶어

⏺ 좋아. 스킬 초안을 작성하기 전에 몇 가지 확인할게.

  저장 위치
  - 프로젝트별로 다른 파일 (<레포>/.claude/memory/mistakes.md)
  - 아니면 전역으로 하나 (~/.claude/mistakes.md)

  "실수"의 범위
  - 사용자가 직접 지적/교정할 때만 기록
  - 아니면 잘못된 파일 수정, 빌드 에러 등 Claude가 스스로 판단할 때도 포함

  트리거 방식
  - 사용자가 /record-mistake 같은 커맨드를 직접 실행
  - 아니면 Claude가 교정 받는 상황을 감지하면 자동으로 기록

  기록 형식
  - 실수 내용 + 올바른 방법 + 컨텍스트(어떤 파일/기능)
  - 다른 형식 원하는 게 있어?

❯  저장 위치: 전역, 실수 범위: 클로드가 스스로 판단할 때도 포함, 트리거 방식: 커맨드 실행하거나 자동 기록 둘다. 기록 형식: 첫번째 내용으로 진행

⏺ 확인됐어. 바로 스킬 초안 작성할게.
⏺ 완료. 생성된 것들:

  lessons/lessons.md — 설치: cp lessons/lessons.md ~/.claude/commands/lessons.md

  동작:
  - 교정 발언, 빌드 실패, 자기 인식 시 ~/.claude/mistakes.md에 자동 기록
  - /lessons 실행하면 누적 실수 목록 요약 출력
  - 세션 시작 시 mistakes.md 읽어 과거 실수 파악

  CLAUDE.md — 세션 시작 규칙에 mistakes.md 로드 추가

  설치하면 바로 동작해. 실수 파일은 전역(~/.claude/mistakes.md)이라 모든 프로젝트에서 공유돼.
#### 리드미 자동 업데이트 스킬 생성
⏺ 어떤 작업을 자동화하고 싶으신가요?
  지금 이 대화에서 반복적으로 하신 작업이 있다면 그걸 스킬로 만들 수도 있고, 아예 새로운 아이디어를 말씀해 주셔도 됩니다.
❯ @CLAUDE.md 의 문서화 부분 전체를 스킬로 만들고 싶은데 쫌 디벨롭 해서 만들고 싶어.   
⏺ CLAUDE.md의 문서화 섹션을 확인했습니다. 현재 내용은:
  - 트리거: 코드 구현 완료, 디렉토리/파일 추가·삭제, 기능 변경, 설치 방법 변경, 에이전트/스킬 변경
  - 대상: 루트 README.md + 해당 디렉토리 README.md 동기화
  - 범위: 디렉토리 구조 트리, 도구 목록 테이블, 설명/설치/사용 방법

  "디벨롭"하고 싶다고 하셨는데, 아래 중 어떤 부분을 발전시키고 싶으신가요?

  기능 확장
  - README 품질 검사 (누락된 섹션, 오래된 내용 감지)
  - 변경된 파일을 자동으로 감지해서 영향받는 README 파악
  - 다국어 README 동기화 (README.md / README.en.md)
  - CHANGELOG 자동 업데이트

  동작 방식 개선
  - 변경 전후 diff 기반으로 최소한의 섹션만 수정
  - 기존 README 스타일/톤을 분석해서 일관성 유지
  - 실제 파일 구조를 스캔해서 트리 자동 생성

  트리거 개선
  - git diff를 분석해서 어떤 문서가 영향받는지 자동 판단
  - 커밋 메시지 기반으로 문서화 범위 결정

어떤 방향이 맞는지, 또는 다른 아이디어가 있으시면 말씀해 주세요!
❯ 트리거, 대상, 범위 내용이 너무 현 레포에 특정지어 작성된거 같고 각 레포의 특징에 맞게 리드미를 수정하고 싶은데 그게 가능한가? 지금 레포의 경우 디렉토리 또는 파일추가/삭제, 설치 방법 변경, 에이전트/스킬 추가 또는 수정에 맞는거고 코드 구현하는 레포에서는 트리거, 범위, 대상 등이 다를 수 있잖아?
⏺ 완전히 가능합니다. 핵심 아이디어는 이거죠:

  ▎ 스킬이 레포를 먼저 분석해서 → 레포 유형에 맞는 트리거/대상/범위를 스스로 결정하고 → 그에 맞게 README를 업데이트

  예를 들면:

  ┌─────────────────────┬─────────────────────────────────────┬───────────────────────────────────────┐
  │      레포 유형      │             트리거 예시             │               범위 예시               │

  ├─────────────────────┼─────────────────────────────────────┼───────────────────────────────────────┤

  확인하고 싶은 것 몇 가지:
  1. 레포 분석 방식 - 스킬이 실행될 때 레포를 자동으로 분석해서 유형을 판단하길 원하나요? 아니면 처음 한 번 사용자가 유형을 지정하면 그걸 기준으로?
  2. 사용 시점 - 이 스킬은 언제 호출되길 원하나요?
    - 사용자가 직접 /document 같은 명령 실행
    - 코드 변경 후 자동으로 Claude가 판단해서 실행
    - 둘 다
  3. 기존 README 스타일 유지 - 이미 README가 있으면 그 스타일/구조를 최대한 유지하면서 업데이트하길 원하나요?

❯ 1. 스킬이 실행될때 분석했음 좋겠고  2. 둘다 였으면 좋겠고 3. 기존 리드미 스타일 유지해줘

⏺ 좋아요. 정리하면 이런 스킬이네요:

  - 실행 시 레포를 분석해서 유형 자동 판단

### skill-creator improvement measure 요청 결과
❯ 스킬의 improvement measure 은 어케해? 예를 들어 /lessons 하기 전후?

⏺ 스킬 개선 측정은 반복(iteration) 비교 방식으로 합니다. /lessons 같은 스킬에 적용하면:

  측정 방식

  1. 테스트 케이스 실행 (with_skill vs baseline)

  - 신규 스킬: with_skill vs without_skill (스킬 없이 같은 프롬프트)
  - 기존 스킬 개선: with_skill (새 버전) vs old_skill (이전 버전 스냅샷)

  두 경우를 동시에 병렬로 실행해서 비교합니다.

  2. Assertions (정량 지표)

  각 테스트 케이스에 객관적으로 검증 가능한 체크리스트를 붙입니다.

  /lessons 예시라면:
  - 실수 내용이 mistakes.md에 기록됐는가?
  - 다음 세션에서 같은 실수를 반복하지 않는가?
  - 기록 형식이 올바른가 (rule / Why / How to apply)?

  3. Benchmark 생성

  iteration-1/benchmark.json
    - with_skill: pass_rate 70%, 23s, 8k tokens
    - without_skill: pass_rate 30%, 18s, 6k tokens

  4. Eval Viewer로 사람이 직접 비교

  generate_review.py로 브라우저에 결과를 띄워서:
  - 스킬 있을 때 / 없을 때 출력을 나란히 보고
  - 피드백 텍스트 남기기
  - iteration 2에선 이전 iteration 결과도 같이 표시

  5. 반복

  피드백 → 스킬 수정 → iteration-2 실행 → 다시 비교
